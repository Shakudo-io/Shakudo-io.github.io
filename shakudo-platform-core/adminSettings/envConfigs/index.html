Environment Configs

An Environment Config encapsulates the desired state of your application's deployment. It allows you to specify the container image, environment variables, compute resources (CPU and memory), and secure mechanisms for injecting sensitive data like API keys or database credentials.

Creating and using Environment Configs ensures that your development environment accurately mirrors the production environment, minimizing discrepancies and issues during deployment.

This consistency enables seamless collaboration among developers, streamlines testing and debugging processes, and ultimately facilitates a smoother transition from development to production deployments.

You can find the Environment Config page on the left panel of the UI:

Create Custom Environment Configs​

Prerequisities​

These are a few requirements when creating an Environment Config:

  • An Image that is accessible to Shakudo.
  • Volumes and volume mounts that are accessible within your cluster.
  • Appropriate node pools or resources that are accessible within your cluster.

Steps to Create an Environment Config​

You can create a new Environment Config for your Sessions, Jobs, or Services.

Follow these steps to do so:

  1. Navigate to the Environment Configs tab and click the + button to create a new Environment Config.

  2. Customize the details. For details on each field, see Environment Config configurations below.

  3. Click CREATE.

  4. To view and use the new Environment Config you must first hard refresh the page.

Environment Config configurations​

General​

Name: The name that will appear when choosing which Environment Config to use in Sessions, Jobs, and Services.

Internal Name: Unique name that acts as the ID for the Environment Config.

Description: The main use of the Environment Config. This description appears in the Environment Config page as well as in the Session Type selection.

Image URL: URL to the Docker Image for the Custom Environment Config.

Resources​

CPU Request: Minimum CPU resources.

CPU Limit: Maximum CPU resources.

Memory Request: Minimum Memory resources.

Memory Limit: Maximum Memory resources.

note

CPU resources are measured in milli-vCPU (m): A virtualized CPU resource allocation unit that represents 1/1000th of a single virtual CPU core. A VM configured with 2000 milli-vCPUs has an equivalent of 2 full vCPU cores allocated.

Memory resources are measured in Gibibytes (Gi): 1 Gi is precisely 2^30 or 1,073,741,824 bytes. A 512 GiB SSD has a capacity of 549,755,813,888 bytes versus a 512 GB drive being 512,000,000,000 bytes. It is a higher accuracy unit than GBs but almost equivalent.

GPU: GPU type.

# of GPU's: Number of GPUs.

note

Please contact the Shakudo team if you require assistance in selecting or provisioning GPUs.

Advanced​

Environment Variables: Use this field to specify the key and values for your Environment Variables.

The remainder of the Advanced settings are not required for standard configurations.

Advanced settings

Edit Volumes: YAML - A volume represents a directory, with data accessible to containers and persists beyond the lifecycle of individual containers.

Example
volumes:
  - name: data-volume
    hostPath:
      path: /data
      type: Directory

Edit Volume Mounts: YAML - A volume mount is a directory within a container's filesystem backed by an external volume, allowing data to persist beyond the container's lifecycle.

Example
volumeMounts:
    - name: data-volume
      mountPath: /usr/share/nginx/html

Edit Tolerations: YAML - Tolerations are a Kubernetes setting that allows a pod to be scheduled onto nodes with specific taints, overriding the default scheduling restrictions.

Example
tolerations:
  - key: node-type
    operator: Equal
    value: edge
    effect: NoSchedule

Node Selector Key: STRING - the key that is used to label nodes in a Kubernetes cluster, allowing for specific pod scheduling based on node characteristics

Node Selector Value: STRING - the value associated with a particular node selector key, used to identify and select nodes with that specific label for pod scheduling

note

For more information see:

  • Kubernetes guide on Volumes.
  • Kubernetes guide on Tolerations.
  • Kubernetes guide on Node Selector.

Check installed Python Packages in an Environment Config​

To view the installed Python packages in an Environment Config, follow these steps from the Sessions pane:

  1. Create a new session.

  2. Select the Environment Config to check and click Create Session.

  3. Open the browser Session and open a Jupyter Notebook.

  1. Type pip freeze to see pip packages installed or conda list to see conda packages installed.

Updating an Environment Config​

You can edit Environment Configs by clicking on the pencil icon from the Environment Config tab. By default, you can only edit Custom Environment Configs.

note

When an Environment Config is edited (and saved), all new jobs and sessions will use the updated version.

All jobs, sessions, and services in progress during the update will continue using the original version.

So, any jobs, sessions, and services in progress need to be restarted for the change to take effect.
